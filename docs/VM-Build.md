# Hyper-V VM Build

This document covers the repeatable Hyper-V VM provisioning process used by this lab. The current proven build creates Windows 11 Enterprise VMs from the `WIN11-WSL2` hardware profile, using a clean ISO install so each machine has unique Hyper-V, Windows, MDE, Entra ID, and Intune identity.

## Recommendation

Do not clone an existing VM that is already onboarded to Microsoft Defender for Endpoint (MDE), Entra ID, or Intune.

A Hyper-V clone can get a new Hyper-V VM ID, but Windows and security-management identity are separate things:

- Hyper-V VM ID: generated by Hyper-V when a new VM is created or imported with a new ID.
- Windows machine SID: changed by Sysprep `/generalize`, or naturally unique from a fresh Windows install.
- MDE device identity: created during Defender onboarding and should not be duplicated by cloning an already onboarded machine.
- Entra ID / Intune identity: created during join/enrollment and should be unique per device.

For this lab, use a clean VM build and let Intune or MDE onboarding policy create the endpoint identity after enrollment.

You also need:

- Enough disk space for the new VHDX.

## Provision A VM

Use the repeatable provisioning entrypoint. It accepts the new VM name, self-elevates when needed, creates the VM, applies nested virtualization and nested networking while the VM is still powered off, then validates the result.

From PowerShell on the host:

```powershell
Set-Location "C:\Users\Lorenzo\Documents\Win11_Intune_Project"
.\scripts\Provision-Win11HyperVVM.ps1 -VMName "WIN11-INTUNE"
```

To create another lab VM from the same source hardware profile, change only the VM name:

```powershell
.\scripts\Provision-Win11HyperVVM.ps1 -VMName "WIN11-APPLKR"
```

The entrypoint reads the hardware configuration from `WIN11-WSL2`, saves it to `templates\<VMName>.hyperv-template.json`, and creates one VM with the requested name. The VM name is carried into the VM folder, VHD file names, log names, report names, and template spec name.

The VM is created with:

- New Hyper-V VM ID.
- Blank virtual disks sized like the source disks.
- VHD file names that begin with the target VM name.
- Same CPU count and memory configuration as the source VM.
- Nested virtualization enabled for WSL, Docker, and nested lab workloads.
- MAC address spoofing enabled on every VM network adapter for nested networking.
- Same generation, checkpoint policy, firmware/Secure Boot settings, TPM setting, integration-service state, network switch, and VLAN configuration as the source VM.
- Windows 11 Enterprise ISO attached for installation.

The entrypoint intentionally uses new dynamic MAC addresses so the machine remains unique.

If you ever want a reusable blank template VM as well, run:

```powershell
.\scripts\Provision-Win11HyperVVM.ps1 -VMName "WIN11-INTUNE" -CreateTemplateVM
```

## Built-In Validation

Provisioning succeeds only if validation passes immediately after creation and before the first VM start. Reports are written to `reports\Provision-<VMName>-<timestamp>.validation.json`, and transcripts are written to `logs\Provision-<VMName>-<timestamp>.log`.

If you run validation later after starting the VM, the powered-off check is expected to fail.

Validation checks include:

- VM exists and remains powered off.
- Hyper-V VM ID differs from the source VM.
- VM generation, memory, CPU count, checkpoint policy, firmware, TPM, integration services, network switch, and VLAN match the source VM.
- Nested virtualization is enabled.
- MAC address spoofing is enabled on every VM network adapter.
- MAC addresses are unique from the source VM.
- VHD files are new, sized like the source disks, and named with the target VM name.
- Windows install ISO is attached.

To validate an existing VM manually from an elevated shell:

```powershell
.\scripts\Test-HyperVVmProvision.ps1 -VMName "WIN11-INTUNE"
```

## Bootstrap Tools Inside A Windows VM

After Windows setup completes, sign in with a local admin account first. Then open Windows PowerShell as Administrator inside the VM and run:

```powershell
Set-ExecutionPolicy Bypass -Scope Process -Force
```

Copy or share this workspace into the VM, then run:

```powershell
.\scripts\Bootstrap-Win11IntuneTools.ps1
```

The bootstrap installs:

- PowerShell 7
- Python 3.13
- Azure CLI
- Az PowerShell module for PowerShell 7
- Azure CLI extensions for Log Analytics and Kusto, where available
- GitHub CLI
- Node.js current release
- Visual Studio Code
- Common VS Code extensions for PowerShell, Python, Azure CLI, Azure Account, and Kusto

## Join A Windows VM To Intune

Before joining, ensure the VM name is correct:

```powershell
Rename-Computer -NewName "WIN11-INTUNE" -Restart
```

After restart:

1. Open Settings.
2. Go to Accounts > Access work or school.
3. Select Connect.
4. Choose Join this device to Microsoft Entra ID.
5. Sign in as `lireland@dibsecurity.onmicrosoft.com`.
6. Complete MFA and enrollment prompts.

Enrollment succeeds only if the user has the required license, MDM user scope is enabled, and enrollment restrictions allow Windows enrollment.

## Verify Enrollment

Inside the VM, run:

```powershell
.\scripts\Verify-Win11Intune.ps1
```

Also check:

```powershell
dsregcmd /status
```

Look for:

- `AzureAdJoined : YES`
- `MdmUrl` populated
- Device visible in Microsoft Intune admin center

## If You Must Clone Instead

Only clone from a generalized, offboarded source:

1. Offboard from MDE.
2. Disconnect from Intune/Entra ID if enrolled or joined.
3. Run Sysprep inside the source VM:

   ```cmd
   C:\Windows\System32\Sysprep\Sysprep.exe /generalize /oobe /shutdown
   ```

4. Export the VM.
5. Import with a new Hyper-V ID:

   ```powershell
   Import-VM -Path "C:\Path\To\Export\Virtual Machines\<vm-id>.vmcx" -Copy -GenerateNewId
   ```

For the MDE/Intune scenario, a clean build is the safer option.
